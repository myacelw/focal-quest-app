## 1. 背景与目标

### 1.1 痛点
儿童"翻转拍"（Accommodative Flipper，双眼调节灵敏度训练）是常见的视觉训练手段。传统做法是隔着翻转拍逐行读卡片上的文字/图案，**视觉刺激单一、反馈枯燥**，孩子极易疲劳、无法坚持。本项目用户家中已有翻转拍，但孩子**"有拍子却坚持不下来"**。

### 1.2 目标
引入游戏化机制（借鉴 Duolingo 的行为塑造模型），把训练变得有节奏、有即时反馈、有成就感，让 **9-12 岁孩子能在家自主坚持**完成日常翻转拍训练。

### 1.3 成功标准
- 孩子愿意主动打开、连续多日坚持训练（留存 > 传统读卡片）。
- 训练流程符合翻转拍训练的基本医学规范（距离、视标大小、单眼交替、翻转节奏）。
- 家长能看到孩子的训练记录和调节力变化趋势。

## 2. 医学背景（训练有效性依据）

> 精简自临床规范，作为功能设计的约束来源。详细文献见关联的 Gemini 调研文档。

- **原理**：翻转拍两面分别是正镜片（Plus，如 +2.00D）和负镜片（Minus，如 −2.00D）。透过正镜片看近处视标时眼睛需**放松**调节，翻到负镜片时需**收缩**睫状肌增加调节。反复交替强迫睫状肌快速舒缩，提升调节灵敏度、幅度和反应速度。
- **计量单位 CPM**（Cycles Per Minute，每分钟周期数）：一个完整 cycle = 一次正镜片看清 + 一次负镜片看清（即两次翻转）。
- **训练规范**：
  - 眼到屏幕**标准距离 40cm**。
  - 视标大小按孩子最佳矫正视力个性化配置（常用 0.8 级别）。
  - **遮盖单眼**，用另一只单眼隔着翻转拍看视标，看清后报答案，然后翻拍看下一个。
  - 单眼持续约 **3 分钟**，然后**换另一只眼**同样练 3 分钟。
- **学龄儿童常模参考**（40cm，±2.00D 翻转拍）：
  - 单眼调节灵敏度（MAF）正常参考区间约 11 cpm 及以上，临界值约 7 cpm 以下为异常。
  - 双眼调节灵敏度（BAF）正常参考区间约 8 cpm 及以上。
  - （以上为参考量级，具体以孩子的眼科医生建议为准。）

## 3. 目标用户与使用场景

### 3.1 用户
- **主用户（受训者）**：9-12 岁小学高年级儿童。识字、发音清楚，能读数字/字母/方向词，能承受有一定深度的游戏。
- **辅助用户（家长）**：负责初始设置、偶尔监督翻拍是否规范、查看统计与趋势。

### 3.2 场景
- 家中安静房间，孩子坐在桌前，iPad 立于约 40cm 处。
- 一只手拿翻转拍、一只手遮住非训练眼——**双手被占**，这是交互设计的最大约束。
- 每天训练一轮（左眼 3 分钟 + 右眼 3 分钟），追求"短、稳、能坚持"。

## 4. 范围与非目标

### 4.1 MVP 范围（做）
| 模块 | 内容 |
|---|---|
| 平台 | **iPad Web 单平台** |
| 交互 | 语音报答案为主 + 触控兜底 |
| 训练驱动 | 节奏引导翻拍（信任式，不检测翻转） |

| 设置 | 视标类型、视力级别、拍子屈光度、单眼时长、音量等 |

### 4.2 非目标（明确不做 / 推迟）
> 这些多来自 Gemini 调研报告，评估后认为对"自家孩子用、快速起步"属**过度设计**，故排除或推迟。记录在此以免反复讨论。

| 排除项 | 原因 | 处置 |
|---|---|---|
| 手机 / 电脑 / 电视端 | 聚焦降低复杂度；手机小屏与 40cm 视标空间冲突 | 远期再议 |
| 原生 App / Flutter | Web 起步最快 | 远期商业化再迁移 |
| 手势识别、物理键盘输入 | 双手被占 / 平板不适用 | 不做 |
| 摄像头 CV：虹膜测距、闭眼(EAR)检测、ArUco 翻转检测 | 重型、算力大、集成复杂；MVP 不需要 | 迭代 3+ 可选 |
| 蓝牙智能翻转拍硬件 | 硬件成本与开发成本高 | 远期 |
| 云同步 / 账号系统 / Supabase | 增加复杂度；国内访问不稳；隐私面扩大 | 迭代 3+ 可选 |
| 回归分析开"训练处方"、疲劳指数自动调参 | 建立在可信数据上，MVP 数据不足够严谨 | 迭代 3+ 可选 |
| 离线 ASR 的复杂 FST 热词优化 | 先验证基础方案是否够用 | 按迭代 0 结果定 |

### 4.3 已接受的代价（重要）
因为**不检测翻转**，软件无法保证孩子真的翻了拍——语音答对 ≠ 真的翻转了屈光度。MVP 的训练有效性依赖：① 节奏引导让翻拍成为自然节拍；② 家长偶尔监督。**这是"家庭级够用"而非"临床级严谨"**，用户已知悉并接受。

## 5. 功能需求

### 5.1 屏幕物理标定（必做，MVP 地基）
- **问题**：不同 iPad 的 DPI/PPI 不同，同样 CSS 像素物理尺寸不同；不标定则视标大小（=视力级别）失去意义。
- **方案**：首次使用/设置里提供标定页——屏幕显示一条可拖动的横条，用户拿一张**标准银行卡（85.6mm 宽）**贴屏对齐，拖到与卡片等宽 → 反算出该屏幕的**像素/毫米（px per mm）**。
- **产出**：保存标定系数，之后所有视标按目标视力级别换算成毫米、再换算成像素显示。
- **验证**：标定后显示一个"应为 20mm"的方块，用户可拿尺子核对。

### 5.2 核心训练循环
一个"训练节" = 单眼 3 分钟（时长可配置）：

1. **准备**：提示遮住指定眼、拍子正镜片面朝眼、保持 40cm（用物理绳/家长监督）。
2. **出题**：屏幕出现视标，按标定后真实物理尺寸显示（对应设定视力级别）。
3. **作答**：孩子看清后**说出答案**（语音）或**点选**（兜底）。
4. **判定 + 反馈**：软件判对错，播放对应音效，当前视标平滑淡出。
5. **翻拍引导**：播放翻转动画 + "咔"提示音，引导孩子翻拍（正↔负面切换）。
6. **新视标过渡淡入**：新视标以"初始轻微高斯模糊 → 逐渐清晰"的过渡出现（模拟睫状肌调节延迟，兼具护眼、防止硬切换视觉不适）。
7. 循环 2-6，实时显示当前 CPM 与得分。
8. **本节结算**：3 分钟到，展示答题数、正确率、平均 CPM、得分。
9. **换眼**：提示遮另一只眼，重复 1-8。
10. **完成打卡**：全部完成后，打卡 +1、更新连续天数与积分，播放奖励音效。

**节奏策略（已确认）**：MVP 用**自由节奏**——孩子答完即引导翻拍、不强制固定节拍，避免跟不上产生挫败。迭代 2 增加可选的**"限时挑战"**模式（固定节拍逐渐加速，用于冲 CPM）。

### 5.3 答题交互
- **主：语音识别**。识别范围高度受限：数字 `1-9` 或方向词 `上/下/左/右`。方案在迭代 0 验证后确定（Web Speech API vs WASM 离线模型）。
- **兜底：触控点选**。屏幕下方给出候选项（如 4 个方向按钮 / 数字键），孩子可在翻拍间隙用另一只手点选。语音识别失败或环境嘈杂时切换。
- 兜底方式应可在设置中设为"始终显示"或"仅语音失败时显示"。

### 5.4 视标系统
- **类型（MVP）**：
  - 数字（1-9）：语音报数字。
  - 方向 E（Tumbling E）/ Landolt C：语音报"上下左右"。
- **大小**：由「目标视力级别 + 40cm 距离 + 屏幕标定系数」共同换算出物理尺寸再显示。
- **可配置**：视标类型、视力级别（默认 0.6-0.8 起步，可调）。
- 视标应随机出题，避免连续重复导致孩子背答案。

### 5.5 音效反馈
- 正确：清脆上行音。
- 错误：温和、非惩罚性的下行提示音。
- 奖励/通关/打卡：金币喷洒类高频特效音。
- 音量可调、可静音。素材优先用免费音效库。

### 5.6 坚持引擎（打卡 / 积分）
- **每日打卡**：完成当日训练（左右眼各一节）即打卡。
- **连续天数（streak）**：连续打卡天数，中断归零。
- **积分**：每次正确答题得基础分；连续打卡有加成系数（如连续越多倍数越高，设上限）。
- 积分用途：MVP 仅累计与展示；兑换/商城放迭代 2+。

### 5.7 数据记录与统计
- **记录维度**（每次训练）：日期时间、眼别、时长、视标类型、视力级别、答题数、答对数、正确率、平均/峰值 CPM、得分。
- **打卡维度**：每日是否完成、连续天数、当日/累计积分。
- **统计展示**：按日 / 周 / 月汇总；用直观图表（折线/柱状）展示训练次数、时长、正确率、CPM 走势。
- **趋势解读（MVP 轻量版）**：展示 CPM 随时间的走势曲线即可；"智能回归分析/处方"推迟到迭代 3+。

> **关于 CPM 的诚实说明**：因不检测翻转，MVP 的 CPM 为**推算式**——由"孩子答对并被引导翻拍"的频率推算（约等于每分钟答对题数 / 2），其准确性依赖孩子诚实翻拍。统计界面应避免把它标榜为临床精确值。

### 5.8 设置
- 视标类型、视力级别、拍子屈光度（默认 ±2.00D，待确认实际度数）、单眼训练时长（默认 3 分钟）、音量、兜底触控显示策略、屏幕标定入口、（可选）训练提醒。

## 6. 交互与体验要求
- **儿童友好**：大按钮、明亮配色、少文字、语音/图形引导；错误反馈不惩罚、鼓励为主。
- **视觉平滑**：视标切换必须过渡（淡出 + 模糊→清晰淡入），杜绝生硬跳变，兼顾护眼。
- **距离/坐姿提醒**：MVP 用静态提示（训练前提醒 40cm、坐正、遮眼）。
- **iPad 适配**：横竖屏择一固定（建议横屏），防误触，支持"添加到主屏"以全屏 PWA 方式运行。

## 7. 技术架构

- **前端**：TypeScript + React + Vite。
- **视标/动画/过渡**：DOM + CSS（含 CSS filter 实现模糊过渡）。**MVP 不引入 Pixi.js/Phaser 游戏引擎**，等迭代 2 做重美术主题关卡时再引入。
- **语音**：迭代 0 验证后二选一：
  - Web Speech API（`webkitSpeechRecognition`）：几行代码、零成本，但走云、需联网、对受限词表无优化，iPad Safari 支持需实测。
  - WASM 离线模型（sherpa-onnx / vosk-browser 等）：离线、可限定词表，但加载模型有体积、iPad 性能需实测。
- **存储**：IndexedDB，使用 Dexie.js 封装。
- **运行/部署**：
  - 开发期：iPad 与开发电脑同一 WiFi，iPad Safari 访问局域网内的 Vite dev server（`https://<电脑局域网IP>:<port>`）——**完全不依赖国外网络**。语音/PWA 需 HTTPS，dev server 启用本地证书。
  - 上线（远期）：Vercel / Cloudflare Pages 静态托管，自家使用国内访问基本够用。
- **素材**：音效用免费库；图片/动画用 GPT/Gemini 生成（Claude 提供提示词）。

## 9. 非功能需求
- **性能**：训练循环流畅无卡顿；视标过渡动画顺滑（目标 60fps 感受）。
- **离线**：核心训练不依赖网络（若语音选 Web Speech API 则该功能需联网，作为选型权衡记录）。
- **隐私**：所有数据本地存储；若涉及麦克风，语音数据不做持久化、不上传（除非 Web Speech API 走云——需在选型时明确告知并权衡）。
- **健壮性**：语音识别失败可无缝切触控；异常不打断训练心流。

## 10. 迭代路线图

### 迭代 0 — 技术验证（当前）
**目标**：用最小代码验证"语音"和"标定"能否在 iPad 上真正可用，避免在错误地基上盖楼。
**产出**：iPad Safari 可打开的极简网页，含两个测试页：
1. **标定页**：拖滑块对齐银行卡宽度 → 算屏幕 px/mm → 显示"应为 20mm"方块供尺子核对。
2. **语音测试页**：屏显"请说 X" → 孩子说数字/方向 → 显示识别结果与延迟；对比 Web Speech API 与一个 WASM 离线方案。

**验收标准**：
- 标定后 20mm 方块实测误差 < 5%。
- 用真实孩子测一轮：识别准确率达可用水平（目标 > 90%）、延迟可接受（< 1s）。
- **明确 MVP 采用哪个语音方案**。

### 迭代 1 — 核心可玩 MVP
完整训练循环 + 屏幕标定 + 视标系统 + 语音/触控交互 + 视觉过渡 + 音效 + 每日打卡/连续天数/积分 + 单节结算 + 基础日/周/月统计。轻美术。

### 迭代 2 — 游戏化增强
勋章体系与勋章墙、随机彩蛋奖励、第一个主题关卡（如太空射击）、限时挑战模式、积分兑换。可能引入 Pixi.js/Phaser。

### 迭代 3+ — 可选增强
更多主题关卡、CPM 趋势智能分析、云同步与多设备、摄像头距离/遮挡/翻转增强、Flutter 原生迁移与应用商店上架。

## 13. 术语表
- **翻转拍 / Flipper**：两面装正负镜片的调节训练工具。
- **调节灵敏度 / Accommodative Facility**：睫状肌快速舒缩、切换焦点的能力。
- **CPM / Cycle**：每分钟周期数；1 cycle = 正镜片看清 + 负镜片看清。
- **视标 / Optotype**：视力表上的字符（数字、E、Landolt C 等）。
- **MAF / BAF**：单眼 / 双眼调节灵敏度。
- **标定 / Calibration**：测算屏幕像素与物理毫米换算关系。
